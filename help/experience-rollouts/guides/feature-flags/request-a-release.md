# Release aanvragen {#request-a-release}

## Vereisten {#prerequisites}

* U hebt de **rol van de Manager van de Versie** — zie <!-- broken link[User roles](../teams/user-roles.md) -->
* Uw toepassing wordt bewaakt - zie [ Op uw toepassing ](../applications/onboard-your-application.md)

>[!TIP]
>
>Reviseer <!--[Releases and cross-team feature groups](releases-and-cross-team-feature-groups.md)--> voordat u een release aanvraagt. Een multiteam functiegroep kan aan uw behoeften met minder overheadkosten voldoen — het is zelf-gediend en steunt rijkere publiek het richten.

## Een ondersteuningsverzoek indienen {#submit}

Maken van vrijgave is geen zelfvoorziening. Contact opnemen met de ondersteuning Rollouts om een nieuwe release aan te vragen. Geef de volgende informatie op:

| Veld | Beschrijving |
|---|---|
| **Naam** | Een unieke naam voor de release. Plaats geen spaties in de naam van de release. |
| **Team** | Het team dat eigenaar zal zijn van de release. |
| **Milieu** | De omgeving waarin de release moet worden gemaakt (werkgebied of productie). |
| **Doelstelling** | Een korte beschrijving van wat deze release zal opleveren en waarom. |
| **Toepassingen** | De toepassing of toepassingen die in de release moeten worden opgenomen. |
| **Duur** | Hoelang moet u de release actief blijven? Verwacht wordt dat de releases binnen drie maanden worden gesloten of gebaseerd. Als u langer nodig hebt, geef dan een zakelijke rechtvaardiging. |

>[!NOTE]
>
>Het aantal actieve releases op een bepaald tijdstip is beperkt. Plan de basislijn of sluit uw versie binnen drie maanden om capaciteit voor andere teams vrij te maken.

## Nadat de release is gemaakt {#after}

Zodra u bevestiging ontvangt dat de versie is gecreeerd, login aan de console en voltooi de versieconfiguratie. Zie [ het werkschema van de Versie van begin tot eind ](release-workflow-end-to-end.md) voor de volledige opeenvolging van stappen.
